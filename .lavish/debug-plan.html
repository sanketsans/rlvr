Goal

Set a breakpoint and inspect tensor shapes with one keystroke, driven by a per-repo config file I can version & tweak.

Now

No DAP extras enabled, no debugpy, no .vscode/launch.json. Breakpoint binding feels like it collides with tmux.

Desired

nvim-dap reads .vscode/launch.json from the repo; F9 toggles a breakpoint with zero ambiguity.

1 Why it “detaches” — keystroke routing

Verified against your dotfiles: ~/.tmux.conf → set -g prefix C-a; nvim leader = Space (LazyVim default, no override). These are different keys — <leader>db cannot detach tmux. The detach comes from the prefix path, easy to trigger because Caps Lock = Control.

tmux prefix path
Caps=CtrlA →tmux grabs prefix →d →DETACH session
nvim leader path
Space →which-key (needs dap.core) →db →toggle breakpoint — never reaches tmux
proposed fix
F9 →straight to nvim-dap →toggle breakpoint — no prefix, no which-key timeout, no tmux

Takeaway: <leader>db isn’t broken — but a single dedicated function key removes the whole class of confusion (and the which-key wait). Decision: bind F9 and disable <leader>db (done via { "<leader>db", false } in the keys spec).

2 Per-project launch.json (the VSCode model, in nvim)

Yes — nvim-dap reads a standard .vscode/launch.json via require("dap.ext.vscode").load_launchjs(). Drop one file per repo; edit args/env there without touching your nvim config. We commit it to rlvr so the tiny-CPU and remote-attach configs travel with the project.

create · rlvr/.vscode/launch.json  ·  two configs: local-CPU launch + remote GPU attach
  • Local CPU (tiny grpo): CUDA_VISIBLE_DEVICES="" forces CPU; --max-steps 2, dataset.max_samples=4 keep an iteration to seconds; justMyCode:false lets you step into torch/transformers when a shape blows up.
  • Attach to Sky pod: connects to debugpy on the H200 through your SSH tunnel; pathMappings maps your laptop repo ↔ ~/sky_workdir so breakpoints line up.

3 nvim glue: F-key maps + load the repo’s launch.json

A small additive plugin file. Using lazy.nvim’s keys table merges with LazyVim’s DAP spec (it does not clobber the dap.core config), so signs and the comment-stripping JSON decoder stay intact.

create · ~/.config/nvim/lua/plugins/dap-extra.lua
conflict-free keys
F9toggle breakpoint
F5start / continue
F10step over
F11step into
⇧F11step out
inspect tensors (when stopped)
<ldr>dutoggle DAP UI (Scopes/Watch)
<ldr>drREPL → x.shape
<ldr>deeval selection inline

4 The loop, once it’s wired

  1. One-time: :LazyExtras → enable dap.core + lang.python; pip install debugpy (and add it to pyproject dev extras so the pod gets it too).
  2. Local shape/logic bug: open a src/qwen3_rlvr/… file → F9 on the line → F5 → pick “RLVR ▸ local CPU”. Runs on CPU, no GPU needed.
  3. Inspect: <leader>du opens Scopes; in the REPL type batch["input_ids"].shape, advantages.dtype, {k:v.shape for k,v in batch.items()}.
  4. GPU-only bug: sky launch -c rlvr-debug-eus2 …, then on the pod run python -m debugpy --listen 0.0.0.0:5678 --wait-for-client scripts/train_grpo.py …, tunnel ssh -L 5678:localhost:5678 rlvr-debug-eus2, then F5 → “attach to Sky pod”.

6 Risks & open questions

🧩
Adapter name mismatch. If load_launchjs maps the wrong type, configs won’t appear. Resolve by running :lua print(vim.inspect(require('dap').adapters)) and matching "type" in the json.
🔁
Double-load of launch.json. load_launchjs appends on every call; we call it once in config. If duplicates appear, guard with a vim.b flag or load on a single FileType python autocmd.
🛰️
SSH host is ephemeral + remoteRoot is a guess. The rlvr-debug-eus2 alias only exists while that Sky cluster is up — a debug-queue cluster may already be gone, so re-sky launch before attaching. Then confirm ssh rlvr-debug-eus2 'echo $HOME; ls ~/sky_workdir' and fix remoteRoot if $HOME isn’t /home/sky.
🧮
Open: CPU + default bfloat16 is slow/patchy. Add --override dtype=float32 for local runs? (need to confirm the exact config key).
